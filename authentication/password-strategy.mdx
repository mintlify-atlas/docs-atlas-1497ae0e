## Overview

Password authentication is the traditional authentication method using email and password credentials. The Ash Coffee API uses **bcrypt** for secure password hashing and provides complete registration, sign-in, and password reset flows.

## Password Strategy Configuration

From the User resource (`lib/ash_coffee/accounts/user.ex`):

```elixir
password :password do
  identity_field :email
  hash_provider AshAuthentication.BcryptProvider

  resettable do
    sender AshCoffee.Accounts.User.Senders.SendPasswordResetEmail
    password_reset_action_name :reset_password_with_token
    request_password_reset_action_name :request_password_reset_token
  end
end
```

<Info>
  The identity field is set to `:email`, which is defined as a case-insensitive string (`ci_string`) for consistent lookups regardless of email casing.
</Info>

## Registration with Password

### Action Definition

The registration action from the source code:

```elixir
create :register_with_password do
  description "Register a new user with a email and password."

  argument :email, :ci_string do
    allow_nil? false
  end

  argument :password, :string do
    description "The proposed password for the user, in plain text."
    allow_nil? false
    constraints min_length: 8
    sensitive? true
  end

  argument :password_confirmation, :string do
    description "The proposed password for the user (again), in plain text."
    allow_nil? false
    sensitive? true
  end

  # Sets the email from the argument
  change set_attribute(:email, arg(:email))

  # Hashes the provided password
  change AshAuthentication.Strategy.Password.HashPasswordChange

  # Generates an authentication token for the user
  change AshAuthentication.GenerateTokenChange

  # validates that the password matches the confirmation
  validate AshAuthentication.Strategy.Password.PasswordConfirmationValidation

  metadata :token, :string do
    description "A JWT that can be used to authenticate the user."
    allow_nil? false
  end
end
```

### API Request

<CodeGroup>
```bash cURL
curl -X POST https://api.ashcoffee.dev/api/json/users/register_with_password \
  -H "Content-Type: application/json" \
  -d '{
    "user": {
      "email": "user@example.com",
      "password": "SecurePass123!",
      "password_confirmation": "SecurePass123!"
    }
  }'
```

```javascript JavaScript
const response = await fetch('https://api.ashcoffee.dev/api/json/users/register_with_password', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    user: {
      email: 'user@example.com',
      password: 'SecurePass123!',
      password_confirmation: 'SecurePass123!'
    }
  })
});

const data = await response.json();
console.log('Token:', data.meta.token);
```
</CodeGroup>

## Sign In with Password

### Action Definition

```elixir
read :sign_in_with_password do
  description "Attempt to sign in using a email and password."
  get? true

  argument :email, :ci_string do
    description "The email to use for retrieving the user."
    allow_nil? false
  end

  argument :password, :string do
    description "The password to check for the matching user."
    allow_nil? false
    sensitive? true
  end

  # validates the provided email and password and generates a token
  prepare AshAuthentication.Strategy.Password.SignInPreparation

  metadata :token, :string do
    description "A JWT that can be used to authenticate the user."
    allow_nil? false
  end
end
```

### API Request

<CodeGroup>
```bash cURL
curl -X POST https://api.ashcoffee.dev/api/json/users/sign_in_with_password \
  -H "Content-Type: application/json" \
  -d '{
    "user": {
      "email": "user@example.com",
      "password": "SecurePass123!"
    }
  }'
```

```javascript JavaScript
const response = await fetch('https://api.ashcoffee.dev/api/json/users/sign_in_with_password', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'

    user: {
      email: 'user@example.com',

  })
});

const data = await response.json();

// Store token for subsequent requests
```

## Password Requirements

The API enforces the following password requirements:

<ParamField path="min_length" type="integer" default="8" required>
  Minimum password length is 8 characters
</ParamField>

<ParamField path="password_confirmation" type="string" required>
  Must match the password field exactly
</ParamField>

<Warning>
  Passwords are marked as `sensitive? true` in the schema, which prevents them from appearing in logs or error messages.
</Warning>

## Password Hashing with Bcrypt

The API uses `AshAuthentication.BcryptProvider` for secure password hashing:

```elixir
hash_provider AshAuthentication.BcryptProvider
```

Bcrypt provides:
- **Salted hashes**: Each password gets a unique salt
- **Adaptive cost**: Configurable work factor to stay ahead of hardware improvements
- **Slow by design**: Intentionally slow to prevent brute-force attacks

The hashed password is stored in the `hashed_password` attribute:

```elixir
attribute :hashed_password, :string do
  allow_nil? false
  sensitive? true
end
```

## Change Password

Users can change their password by providing their current password:

```elixir
update :change_password do
  require_atomic? false
  accept []
  
  argument :current_password, :string, sensitive?: true, allow_nil?: false
  argument :password, :string,
    sensitive?: true,
    allow_nil?: false,
    constraints: [min_length: 8]
  argument :password_confirmation, :string, sensitive?: true, allow_nil?: false

  validate confirm(:password, :password_confirmation)
  validate {AshAuthentication.Strategy.Password.PasswordValidation,
            strategy_name: :password, password_argument: :current_password}
  change {AshAuthentication.Strategy.Password.HashPasswordChange, strategy_name: :password}
end
```

### API Request

```bash
curl -X PATCH https://api.ashcoffee.dev/api/json/users/:id/change_password \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -d '{
    "user": {
      "current_password": "OldPass123!",
      "password": "NewSecurePass456!",
      "password_confirmation": "NewSecurePass456!"
    }
  }'
```

<Info>
  When a user changes their password, the `log_out_everywhere` add-on automatically revokes all existing tokens for security. The user must sign in again with the new password.
</Info>

## Password Reset Flow

For users who forgot their password, see the complete password reset flow:

<Steps>
  <Step title="Request Password Reset Token">
    User requests a password reset by providing their email
  </Step>
  
  <Step title="Receive Reset Email">
    An email with a password reset link is sent via `SendPasswordResetEmail` sender
  </Step>
  
  <Step title="Reset Password">
    User clicks the link and provides a new password
  </Step>
  
  <Step title="Auto Confirmation">
    Password reset automatically confirms the email address (no separate confirmation needed)
  </Step>
</Steps>

See the [Email Confirmation](/authentication/email-confirmation) guide for the complete password reset implementation.

## Error Handling

Common authentication errors:

<AccordionGroup>
  <Accordion title="Incorrect email or password">
    ```json
    {
      "errors": [{
        "code": "authentication_failed",
        "title": "Authentication Failed",
        "detail": "Incorrect email or password"
      }]
    }
    ```
    Status: `401 Unauthorized`
  </Accordion>
  
  <Accordion title="Password too short">
    ```json
    {
      "errors": [{
        "code": "invalid",
        "field": "password",
        "title": "Invalid",
        "detail": "must have length of at least 8"
      }]
    }
    ```
    Status: `422 Unprocessable Entity`
  </Accordion>
  
  <Accordion title="Password confirmation mismatch">
    ```json
    {
      "errors": [{
        "code": "invalid",
        "field": "password_confirmation",
        "title": "Invalid",
        "detail": "confirmation did not match value"
      }]
    }
    ```
    Status: `422 Unprocessable Entity`
  </Accordion>
  
  <Accordion title="Email already taken">
    ```json
    {
      "errors": [{
        "code": "invalid",
        "field": "email",
        "title": "Invalid",
        "detail": "has already been taken"
      }]
    }
    ```
    Status: `422 Unprocessable Entity`
  </Accordion>
</AccordionGroup>
