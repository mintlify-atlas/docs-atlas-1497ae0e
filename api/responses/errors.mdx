## Overview

The Ash Coffee API follows the [JSON:API specification](https://jsonapi.org/format/#errors) for error responses. All errors return a consistent structure that includes status codes, error titles, and detailed messages to help you diagnose and resolve issues.

## Error Response Format

All error responses use the `application/vnd.api+json` content type and follow this structure:

```json
{
  "errors": [
    {
      "status": "422",
      "title": "Validation Error",
      "detail": "Email has already been taken",
      "source": {
        "pointer": "/data/attributes/email"
      }
    }
  ]
}
```

### Error Object Structure

<ResponseField name="errors" type="array" required>
  An array of error objects. Multiple errors may be returned in a single response.
  
  <Expandable title="Error object properties">
    <ResponseField name="status" type="string" required>
      The HTTP status code as a string (e.g., "400", "422", "500").
    </ResponseField>
    
    <ResponseField name="title" type="string" required>
      A short, human-readable summary of the error type.
    </ResponseField>
    
    <ResponseField name="detail" type="string" required>
      A detailed explanation of the specific error that occurred.
    </ResponseField>
    
    <ResponseField name="source" type="object">
      An object containing references to the source of the error.
      
      <Expandable title="Source properties">
        <ResponseField name="pointer" type="string">
          A JSON Pointer ([RFC6901](https://tools.ietf.org/html/rfc6901)) to the associated entity in the request document (e.g., "/data/attributes/email").
        </ResponseField>
        
        <ResponseField name="parameter" type="string">
          A string indicating which URI query parameter caused the error.
        </ResponseField>
      </Expandable>
    </ResponseField>
    
    <ResponseField name="code" type="string">
      An application-specific error code.
    </ResponseField>
    
    <ResponseField name="meta" type="object">
      Additional metadata about the error.
    </ResponseField>
  </Expandable>
</ResponseField>

## Common HTTP Status Codes

The API uses standard HTTP status codes to indicate the success or failure of requests:

| Status Code | Title | Description |
|-------------|-------|-------------|
| `400` | Bad Request | The request is malformed or contains invalid JSON |
| `401` | Unauthorized | Authentication credentials are missing or invalid |
| `403` | Forbidden | The authenticated user lacks permission to perform the action |
| `404` | Not Found | The requested resource does not exist |
| `422` | Unprocessable Entity | Validation errors or business logic constraints failed |
| `500` | Internal Server Error | An unexpected error occurred on the server |

## Error Examples

### Validation Errors

Validation errors occur when the request data doesn't meet the required constraints. Multiple validation errors may be returned simultaneously.

<CodeGroup>
```json Response (422)
{
  "errors": [
    {
      "status": "422",
      "title": "Validation Error",
      "detail": "Email has already been taken",
      "source": {
        "pointer": "/data/attributes/email"
      }
    },
    {
      "status": "422",
      "title": "Validation Error",
      "detail": "Password must be at least 8 characters",
      "source": {
        "pointer": "/data/attributes/password"
      }
    }
  ]
}
```

```bash Request Example
curl -X POST http://localhost:4000/api/json/auth/register \
  -H "Content-Type: application/vnd.api+json" \
  -d '{
    "data": {
      "type": "user",
      "attributes": {
        "email": "existing@example.com",
        "password": "short",
        "password_confirmation": "short"
      }
    }
  }'
```
</CodeGroup>

### Authentication Errors

Authentication errors occur when credentials are invalid or the authentication token is missing or expired.

<CodeGroup>
```json Response (401)
{
  "errors": [
    {
      "status": "401",
      "title": "Unauthorized",
      "detail": "Invalid email or password"
    }
  ]
}
```

```bash Request Example
curl -X POST http://localhost:4000/api/json/auth/sign_in \
  -H "Content-Type: application/vnd.api+json" \
  -d '{
    "data": {
      "type": "user",
      "attributes": {
        "email": "user@example.com",
        "password": "wrong_password"
      }
    }
  }'
```
</CodeGroup>

### Missing Token Error

When accessing protected endpoints without a valid authentication token:

<CodeGroup>
```json Response (401)
{
  "errors": [
    {
      "status": "401",
      "title": "Unauthorized",
      "detail": "Authentication required"
    }
  ]
}
```

```bash Request Example
curl -X GET http://localhost:4000/api/json/users/me \
  -H "Content-Type: application/vnd.api+json"
```
</CodeGroup>

### Authorization Errors

Authorization errors occur when the authenticated user doesn't have permission to access a resource.

<CodeGroup>
```json Response (403)
{
  "errors": [
    {
      "status": "403",
      "title": "Forbidden",
      "detail": "You do not have permission to perform this action"
    }
  ]
}
```

```bash Request Example
curl -X PATCH http://localhost:4000/api/json/users/other-user-id \
  -H "Content-Type: application/vnd.api+json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -d '{
    "data": {
      "type": "user",
      "id": "other-user-id",
      "attributes": {
        "email": "newemail@example.com"
      }
    }
  }'
```
</CodeGroup>

### Password Confirmation Mismatch

When password and password confirmation don't match during registration or password change:

<CodeGroup>
```json Response (422)
{
  "errors": [
    {
      "status": "422",
      "title": "Validation Error",
      "detail": "Password confirmation does not match password",
      "source": {
        "pointer": "/data/attributes/password_confirmation"
      }
    }
  ]
}
```

```bash Request Example

  -H "Content-Type: application/vnd.api+json" \
  -d '{

      "type": "user",
      "attributes": {
        "email": "user@example.com",
        "password": "secure_password_123",

  }'
```

</CodeGroup>

## Best Practices

<Accordion title="Handle Multiple Errors">
  Always expect the `errors` array to contain multiple error objects. Iterate through all errors to provide comprehensive feedback to users.
  
  ```javascript
  const response = await fetch('/api/json/auth/register', {
    method: 'POST',
    headers: { 'Content-Type': 'application/vnd.api+json' },
    body: JSON.stringify(data)
  });
  
  if (!response.ok) {
    const { errors } = await response.json();
    errors.forEach(error => {
      console.error(`${error.title}: ${error.detail}`);
      // Display error to user based on source.pointer
    });
  }
  ```
</Accordion>

<Accordion title="Use Error Source for Field-Specific Feedback">
  The `source.pointer` field indicates which input field caused the error. Use this to display validation errors next to the appropriate form fields.
  
  ```javascript
  errors.forEach(error => {
    if (error.source?.pointer) {
      // Extract field name from JSON pointer
      // e.g., "/data/attributes/email" -> "email"
      const field = error.source.pointer.split('/').pop();
      showFieldError(field, error.detail);
    }
  });
  ```
</Accordion>

<Accordion title="Check HTTP Status Codes">
  Use the HTTP status code to determine the general category of error:
  - `4xx` codes indicate client errors (invalid input, authentication issues)
  - `5xx` codes indicate server errors (retry the request or contact support)
  
  ```javascript
  if (response.status >= 500) {
    // Server error - retry with exponential backoff
    showErrorMessage('Something went wrong. Please try again.');
  } else if (response.status === 422) {
    // Validation error - show field-specific errors
    displayValidationErrors(errors);
  } else if (response.status === 401) {
    // Authentication error - redirect to login
    redirectToLogin();
  }
  ```
</Accordion>

<Accordion title="Log Errors for Debugging">
  Always log the complete error response during development to understand the full context of errors.
  
  ```javascript
  if (!response.ok) {
    const errorData = await response.json();
    console.error('API Error:', {
      status: response.status,
      errors: errorData.errors
    });
  }
  ```
</Accordion>
